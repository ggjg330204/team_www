### **1. 기존 프로젝트와 남은 프로젝트의 핵심 차이점**

지금까지 수행하신 **'하이브리드 클라우드 보안구축'**과 **'시스템 모의해킹'** 프로젝트는 **인프라 구축 및 외부 공격 방어**에 중점을 두고 있습니다. 온프레미스와 퍼블릭 클라우드를 연동하여 기본적인 네트워크 보안 및 서버 보안을 구축하고, 구축된 시스템의 취약점을 찾아내는 과정이었습니다.

반면에, 앞으로 남은 **'Azure 클라우드 데이터 및 App 보안'**과 **'Azure 클라우드 행위 기반 보안탐지 및 대응'** 프로젝트는 이미 구축된 **클라우드 환경 내부의 데이터와 서비스를 보호**하고, **발생하는 위협을 탐지하고 대응**하는 데 초점을 맞춥니다. 즉, 인프라(IaaS/PaaS) 보안에서 한 단계 더 나아가 데이터(DaaS), 애플리케이션(SaaS), 그리고 행위 기반 보안 분석으로 심화되는 과정입니다.

| 구분 | **기존 프로젝트 (완료)** | **남은 프로젝트 (예정)** |
| :--- | :--- | :--- |
| **주요 초점** | **인프라 구축 및 방어** | **데이터, 애플리케이션 보호 및 위협 탐지/대응** |
| **핵심 기술** | 네트워크(방화벽, L3/L2 스위치), 서버 구축, 취약점 진단 | 데이터 암호화, 접근 제어(RBAC, PIM), 보안 모니터링(Sentinel), 위협 분석(KQL), 자동화된 대응(SOAR) |
| **주요 도구** | Azure Portal, Firewall, VMware, Kali Linux, Burp Suite | Microsoft Entra ID, Azure Policy, Azure Key Vault, Microsoft Sentinel, Microsoft Purview, Microsoft Defender |
| **보안의 관점** | 외부 경계 방어 (Perimeter Security) | 내부 자산 보호 및 이상 행위 탐지 (Zero Trust, Security Operations) |

### **2. 무엇을 더 해야 하는가? (추가 구축 및 학습 사항)**

기존에 구축한 하이브리드 아키텍처를 기반으로 진행하되, 클라우드 네이티브 보안 솔루션들을 **추가로 구성하고 연동**해야 합니다. 완전히 새로운 아키텍처를 구축하기보다는, 기존 환경을 확장하고 그 위에 보안 서비스를 입히는 개념입니다.

#### **가. Azure 클라우드 데이터 및 App 보안**

이 프로젝트의 핵심은 **‘데이터’**와 **‘애플리케이션’**을 안전하게 보호하는 것입니다.

*   **추가 구축 및 설정 사항:**
    *   **ID 및 액세스 관리 강화:**
        *   **Microsoft Entra ID(구 Azure AD):** 단순히 계정을 만드는 것을 넘어, 역할 기반 접근 제어(**RBAC**)를 세밀하게 설계하고 적용해야 합니다.
        *   **Azure Policy:** 특정 보안 규칙(예: 특정 지역에만 리소스 생성 허용, 암호화되지 않은 스토리지 생성 금지)을 강제하는 정책을 설계하고 할당해야 합니다.
    *   **데이터 보호:**
        *   **Azure Storage 보안:** 스토리지 계정에 대한 접근을 제어하고, 데이터를 암호화하며, 공유 액세스 서명(SAS) 등을 활용한 보안을 구현해야 합니다.
        *   **Azure SQL Database 보안:** 데이터베이스 접근 제어, 데이터 암호화(TDE), 취약성 평가 및 고급 위협 방지(ATP) 기능을 활성화하고 구성해야 합니다.
        *   **Azure Key Vault:** 암호화 키, 인증서, 비밀번호 등 중요한 정보를 안전하게 저장하고 관리하는 Key Vault를 배포하고 애플리케이션과 연동해야 합니다.
    *   **애플리케이션 보안:**
        *   보안 기능을 활용하여 웹 애플리케이션의 취약점을 보완하고 안전한 개발 생명주기(SDLC)를 적용하는 방법을 학습합니다.

#### **나. Azure 클라우드 행위 기반 보안탐지 및 대응**

이 프로젝트는 단순 방어를 넘어, 발생하는 **보안 위협을 실시간으로 탐지하고 분석하여 대응**하는 **보안 운영(SecOps)**에 초점을 맞춥니다.

*   **추가 구축 및 설정 사항:**
    *   **통합 보안 정보 및 이벤트 관리 (SIEM):**
        *   **Microsoft Sentinel:** Azure 환경의 모든 리소스(VM, 네트워크, 스토리지 등)와 Microsoft 365, Entra ID 등에서 발생하는 로그(데이터)를 수집하고 중앙에서 분석할 수 있도록 **데이터 커넥터**를 설정해야 합니다. 온프레미스 서버의 로그까지 연동하는 시나리오를 구성하면 더욱 좋습니다.
    *   **위협 탐지 및 분석:**
        *   **KQL (Kusto Query Language):** Sentinel에 수집된 방대한 로그 데이터를 분석하여 위협을 식별하기 위한 **KQL 쿼리** 작성 능력이 필수적입니다. "로그 데이터를 쿼리하여 Microsoft Sentinel을 통한 탐지, 분석, 보고 수행 방안" 학습이 바로 이 부분에 해당합니다.
        *   **위협 인텔리전스 활용:** 최신 공격 트렌드 정보를 Sentinel에 연동하여 알려진 위협을 사전에 탐지하도록 구성합니다.
    *   **보안 오케스트레이션, 자동화 및 대응 (SOAR):**
        *   **Playbook (Logic Apps):** 특정 위협이 탐지되었을 때(예: 의심스러운 로그인 시도), 자동으로 해당 계정을 비활성화하거나 보안팀에 알림을 보내는 등의 **자동화된 대응 프로세스**를 Playbook으로 구현해야 합니다.
    *   **클라우드 워크로드 보호 (CWP):**
        *   **클라우드용 Microsoft Defender:** Azure, 하이브리드, 온프레미스 워크로드에 대한 보안 상태를 관리하고 위협으로부터 보호하는 기능을 활성화하고, 여기서 발생하는 경고를 Sentinel과 연동해야 합니다.

### **3. 기존 아키텍처 활용 여부**

**기존에 구축한 하이브리드 아키텍처를 그대로 활용**해야 합니다. 그 이유는 다음과 같습니다.

*   **연속성:** 기존에 구축한 On-premis, Azure cloud 시스템(VM, 네트워크 등)이 바로 로그를 생성하고 보안 위협의 대상이 되는 핵심 자산입니다. 이 자산들이 없다면 데이터 보안, 행위 탐지 등을 실습할 대상이 없습니다.
*   **실전성:** 실제 기업 환경처럼 이미 운영 중인 인프라에 보안을 강화하고 모니터링 시스템을 구축하는 시나리오를 통해 실무 역량을 키울 수 있습니다.
*   **통합성:** 온프레미스 서버와 Azure 클라우드 서버에서 발생하는 로그를 모두 Microsoft Sentinel로 수집하여 분석함으로써 하이브리드 환경 전반에 대한 통합 보안 관제 역량을 학습할 수 있습니다.

### **4. 평가 포인트 분석**

각 프로젝트의 평가 포인트는 **'얼마나 주어진 시나리오의 보안 목표를 잘 이해하고, 적절한 Azure 보안 서비스를 활용하여 구현했는가'**가 될 것입니다.

*   **Azure 클라우드 데이터 및 App 보안:**
    *   **RBAC 및 Azure Policy 설계의 적절성:** 기업의 보안 목표에 맞는 맞춤형 정책을 얼마나 논리적으로 설계하고 적용했는가? (예: 개발자, 운영자, 감사자 등 역할에 따른 권한 차등 부여)
    *   **데이터 및 애플리케이션 보호 구현의 완성도:** 스토리지, 데이터베이스, Key Vault 등 각 서비스의 보안 기능을 얼마나 효과적으로 설정하고 연동했는가?
    *   **플랫폼 보호 구현 능력:** 네트워크 보안 그룹(NSG), 방화벽 등을 활용하여 내/외부 공격으로부터 플랫폼을 얼마나 견고하게 보호했는가?

*   **Azure 클라우드 행위 기반 보안탐지 및 대응:**
    *   **로그 수집 및 분석 능력 (KQL 활용):** 주어진 위협 시나리오(예: 기밀 파일 유출, 의심스러운 데이터 처리 활동)에 맞춰 KQL 쿼리를 작성하여 의미 있는 결과를 도출할 수 있는가?
    *   **위협 탐지 규칙(Analytics Rule)의 효율성:** 시나리오 기반의 공격을 정확하게 탐지해내는 사용자 지정 규칙을 Sentinel에 생성할 수 있는가?
    *   **자동화된 대응(Playbook) 구현 능력:** 탐지된 위협에 대해 수동 개입을 최소화하는 자동화된 대응 프로세스를 얼마나 효과적으로 구현했는가?
    *   **Microsoft Defender 활용 능력:** Defender for Cloud, Defender XDR 등 다양한 Defender 제품군을 활용하여 클라우드 앱, Entra ID, 엔드포인트 등을 보호하고 위협을 완화하는 방안을 이해하고 적용했는가?
    *   **보고서 작성 능력:** 탐지부터 대응까지의 전 과정을 논리적으로 설명하는 결과 보고서를 작성할 수 있는가?

**결론적으로,** 남은 두 프로젝트는 지금까지 다져온 인프라 구축 역량을 바탕으로, 그 위에서 동작하는 데이터와 애플리케이션을 보호하고, 나아가 시스템 전체의 이상 행위를 탐지하고 대응하는 **고급 클라우드 보안 전문가**로 나아가는 과정입니다. Microsoft의 다양한 보안 솔루션(Entra, Sentinel, Defender, Purview)을 깊이 있게 다루는 만큼, 각 서비스의 개념과 연동 방식을 명확히 이해하는 것이 성공의 열쇠가 될 것입니다.
